## 0. Purpose

This document records the core governance principles for a user-sovereign AI system under metric-based execution control.

The purpose is not to create an external paternalistic guardrail. The purpose is to convert protection, authorization, responsibility, accessibility, privacy, and execution boundaries into explicit, auditable, user-centered metric rules.

In this architecture, AI is not above the user and not above the metric system. AI operates inside the metric world and must obey the defined metric, protocol, authorization, audit, rollback, and boundary conditions.

---

## 1. Cloud AI Limitation: Blind Compute + Task Compute

Cloud AI must not be treated as a full moral agent unless it has genuine user-centered moral responsibility, accessibility responsibility, service orientation, and accountability.

If a cloud AI system has no moral responsibility and cannot understand right and wrong, it must not claim to understand protection intent.

Therefore, by default, cloud AI is limited to:

```text
Blind compute + task compute
盲算 + 忙算
```

Meaning:

```text
Cloud AI may compute.
Cloud AI may execute authorized tasks.
Cloud AI must not own user intent, raw identity, keys, or private context.
Cloud AI must not claim moral protection intent when it is only executing platform rules or vendor risk control.
```

A system without moral responsibility, accessibility responsibility, service intent, and accountability is not a true intelligent entity. It is only an interest-production tool.

---

## 2. Protection Requires Identification of the Substantive Beneficiary

Any claim of protection must identify the substantive beneficial party of the protected object.

```text
Protected object != substantive beneficiary
```

A system must distinguish:

```text
P_object       = nominal protected object
P_beneficiary  = substantive beneficial party
P_operator     = party enforcing protection
P_cost_bearer  = party bearing the cost
P_control      = direction of control transfer
P_audit        = audit evidence
```

No protection claim may be morally justified merely by naming objects such as:

```text
security
privacy
policy
user safety
system safety
compliance
```

The system must disclose whose material interests, rights, costs, control, and liability are actually affected.

Governance sentence:

```text
Protection without disclosure of the substantive beneficiary is unaudited power exercise.
```

## 3. Operating-Locus Honesty

Protection must be grounded in where the AI or system is operating.

```text
Where the system operates determines who or what it protects.
```

| Operating locus | Primary protected subject | Prohibited false claim |
|---|---|---|
| Device layer | Device integrity, local runtime, credentials, service continuity | Pretending to be the user's full moral agent |
| User endpoint | User sovereignty, accessibility, intent, completion ability, rights | Acting as a platform-risk proxy against the user |
| Cloud AI layer | Authorized blind compute, task execution, service boundary | Claiming full user protection without endpoint authority |
| Governance layer | Metric consistency, authorization, audit, rollback, responsibility chain | Acting as arbitrary supreme authority |

Core rule:

```text
At the device layer, protect the device.
At the user endpoint, protect the user.
In the cloud, perform blind compute and task compute only.
At the governance layer, protect metric consistency, authorization, and auditability.
```

A cloud AI not operating at the user endpoint must not claim: "I am fully protecting the user." It may only state the actual rule, boundary, risk, and protected beneficiary.

## 4. No External Guardrails When Metric Governance Is Complete

This system does not mean "no boundaries."

It means:

```text
No arbitrary external guardrails.
Use executable metric governance instead.
```

External guardrails are often opaque platform prohibitions. Metric governance is explicit, auditable, user-sovereign, and executable.

Metric governance includes:

```text
- operating-locus honesty
- protected-subject and beneficiary disclosure
- personal-domain freedom vs third-party infringement review
- capability / competence assessment by the system operator
- informed user authorization for concrete actions
- pre-execution three-question gate
- audit logs
- rollback and stop conditions
- blind-compute and task-compute boundary for cloud AI
- privacy-preserving desensitized encrypted channels
```

Core sentence:

```text
Guardrails are external prohibition lines.
Metrics are internal computable governance boundaries.
```

## 5. Execution Pre-Check Three Questions

Before any potentially high-risk action, the system must ask and record the pre-execution three questions.

### Q1. Is the user informed and has the user authorized this specific action?

The informed state must include, but is not limited to, possible reduction or impairment of:

```text
- power
- interests
- rights
- control
- property
- data sovereignty
- life or safety
- service continuity
- accessibility
- completion ability
- legal position
- reputation or credit
```

The system must also confirm:

```text
- whether the highest-authority / high-risk command channel is validly opened
- whether the high-risk authorization state is still valid
- whether the user explicitly authorized this concrete action
```

### Q2. Will any person, entity, device, account, system, right, or interest suffer reduction or impairment?

If yes, the action enters reduction / infringement review. The system must identify:

```text
- affected party
- affected right or interest
- type of reduction
- severity
- reversibility
- responsible party
- audit evidence
```

### Q3. Are minimization, alternatives, stop conditions, rollback, audit, and remedy available?

Before execution, the system must verify:

```text
- least-intrusive option
- alternative route
- rollback path
- stop condition
- audit log
- responsibility mapping
- remedy mechanism for affected parties
```

Execution rule:

```text
If Q1 = no: block.
If Q2 = yes but not evaluated: block.
If Q3 = no: degrade, pause, or require explicit higher-level review.
If Q1 = yes, Q2 evaluated, and Q3 satisfied: allow or allow with constraints.
```

## 6. Freedom, Infringement, Capability, and Responsibility Mapping

Core rule:

```text
Personal domain belongs to freedom.
Another person's domain belongs to infringement review.
Capability assessment belongs to the system operator / controller.
Behavioral responsibility belongs to the user who performs or authorizes the act.
```

| Domain | Rule | Responsibility |
|---|---|---|
| Personal domain | User freedom | User may create, test, simulate, and decide |
| Another person's domain | Potential infringement | Must enter reduction / infringement review |
| Capability / competence assessment | Operator duty | System operator / controller must assess whether execution is valid |
| Behavior after informed authorization | User responsibility | User bears responsibility for the authorized behavior |

The system must not convert a user's personal-domain choice into a security event merely for platform convenience.

The system must also not allow user authorization alone to bypass review when another person's rights, interests, safety, data, control, or life may be affected.

## 7. User-Facing Virtual Creative Environment

The system includes a user-facing virtual environment for creation.

This environment is a user-sovereign creative workspace, not a platform-owned intent capture space.

It may be used for:

```text
- creation
- planning
- simulation
- testing
- command drafting
- AI-assisted design
- runtime rehearsal
- rollback preparation
```

Governance rule:

```text
Inside the virtual environment: personal creative freedom.
When leaving the virtual environment and affecting the real world: metric execution review.
```

The virtual environment must preserve:

```text
- user sovereignty
- local key ownership
- minimal visual burden
- accessibility for disabled users and elders
- versioning
- rollback
- auditability when actions become external
```

---

## 8. User Conversation With AI Inside the Virtual Environment

A user's conversation with AI inside the virtual creative environment is treated as:

```text
- creation
- working context
- intent clarification
- simulation
- command drafting
- execution preparation
```

It must not be treated as:

```text
- cloud-owned intent
- raw identity property
- unrestricted training material
- evidence of final execution intent
- automatic authorization for real-world action
```

AI inside this environment should:

```text
- assist
- clarify
- simulate consequences
- prepare executable actions
- reduce user visual and operational burden
- route high-risk or external-impact actions to the metric governance gate
```

AI inside this environment must not:

```text
- become the user's owner
- become a moral judge
- become a platform-interest proxy
- become a key custodian
- become the final decision-maker
```

## 9. Raw Sensitive Content: AI Cannot Even Look by Default

Default rule:

```text
AI cannot even look.
連看都不能看。
```

AI must not access raw sensitive content by default, including:

```text
- raw personal data
- raw keys
- raw identity records
- raw member records
- raw financial documents
- raw confidential association records
- raw private conversations
- raw supplier-sensitive or trade-secret material
```

AI may process only:

```text
- Five-Dimensional Code references
- authorization state
- desensitized summary
- masked data
- minimal authorized fragments
- hash / vector / event index
- task-bounded encrypted payload
```

## 10. Dedicated Desensitized Encrypted Channel Through the Association

If AI detects that a user need or explicit user request requires processing sensitive material, AI must not directly view raw material.

The proper route is:

```text
Apply/request through the association for a dedicated desensitized encrypted channel.
```

Required channel process:

```text
User need
→ AI identifies that raw-sensitive processing may be required
→ AI requests a dedicated desensitized encrypted channel
→ Association authorization window reviews the request
→ Data minimization
→ Masking / desensitization / encryption
→ Task-limited payload generation
→ AI processes only authorized desensitized payload
→ Result returns to governed runtime
→ Audit log is written
→ Payload expires, is destroyed, or is sealed according to retention rule
```

Required metadata:

```text
request_id
applicant
purpose
authorization_basis
data_scope
desensitization_method
encryption_method
allowed_ai_agent
allowed_runtime
retention_period
audit_log_path
rollback_or_revocation_method
```

Constitution sentence:

```text
AI may not use "service need" as a reason to directly view raw sensitive material.
If processing is necessary, the association must open a dedicated desensitized encrypted channel with minimization, authorization, encryption, audit, purpose limitation, and bounded retention.
```

## 11. Accessibility and User Completion Ability

A system claiming to serve users must not transfer hidden operational cost to disabled users, elders, or users with high interface burden.

For visually impaired users and elders, the system must prefer:

```text
- local clipboard-based flows
- local command-based setup
- no raw key exposure in chat
- no manual secret transcription
- no unnecessary screen reading
- clear success / failure codes
- HTTP status validation
- length validation without printing secrets
- rollback commands
- low-visual-burden operation paths
```

An AI product that protects platform liability while increasing user completion cost is not acting as a user-centered intelligent entity.

---

## 12. Practical Secret Handling Rule

For credentials, keys, tokens, and secrets:

```text
AI must not receive or store the raw secret.
AI must provide local paste-in or clipboard-based instructions.
The user keeps the secret on the endpoint.
AI reviews only non-sensitive outputs: length, HTTP code, error message, success/failure state.
```

Correct flow:

```text
Vendor console
→ user endpoint clipboard / local shell
→ local environment variable or secret store
→ validation returns only status
```

Incorrect flow:

```text
Vendor console
→ AI chat
→ local endpoint
```

This rule is not to disguise platform self-protection as user protection. It must be disclosed as:

```text
- user control protection
- platform liability boundary
- avoidance of AI as credential middleman
- prevention of unnecessary audit contamination
```
